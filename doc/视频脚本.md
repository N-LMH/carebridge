# CareBridge / 医路桥 视频脚本

## 目标

制作一支 `5-7 分钟` 的黑客松演示视频，重点展示：

- 问题真实
- 产品完整
- 技术可落地
- 医疗边界清晰

## 视频结构

### 1. 开场问题 0:00 - 0:40

建议话术：

“在基层和偏远地区，很多患者真正遇到的问题，不只是看病资源少，而是在去医院之前就不知道自己该怎么办。  
他们不知道症状是否紧急，不知道该挂什么科，也往往不能准确描述病情。  
CareBridge / 医路桥，就是为了解决这个‘正式就医前的信息断层’而设计的。”

### 2. 产品定位 0:40 - 1:10

建议话术：

“CareBridge 是一个 AI 分诊与就医准备助手。  
它不替代医生做诊断，而是帮助患者更早判断风险、更完整整理病情，并生成医生可直接阅读的病情摘要。”

### 3. 演示首页与核心能力 1:10 - 1:50

操作建议：

- 展示首页 Hero 区
- 简要展示右侧 demo presets 和 caseboard
- 强调这个产品已经有完整 workflow，而不是单页原型

建议话术：

“这是产品首页。用户可以直接输入症状，也可以加载一个真实病例预设进行演示。  
右侧还会保留最近分诊记录，方便继续跟进同一位患者的后续变化。”

### 4. 主流程演示 1:50 - 3:50

建议使用预设病例：
`Respiratory watch case`

操作流程：

1. 点击 `Load case`
2. 展示自动填充的患者信息
3. 点击 `Continue assessment`
4. 回答缺失追问
5. 点击 `Generate guidance`
6. 展示结果页

建议话术：

“我们先加载一个典型病例：一位县域患者，已经发热三天，并伴随咳嗽和胸闷。  
系统先接收自然语言和基础病史，然后只追问真正会影响风险判断的几个问题，比如呼吸困难程度、最高体温以及症状是否正在加重。  
完成后，系统会给出风险等级、下一步行动建议、推荐科室，以及为什么得出这个结论。”

### 5. 医生摘要与随访 3:50 - 5:10

操作建议：

- 展示 visit summary 区块
- 点击 `Copy` / `Download` / `Print`
- 填写 follow-up
- 保存一条记录
- 展示 recent caseboard 已更新

建议话术：

“除了分诊建议，CareBridge 还会生成一份可直接给医生看的摘要。  
这份摘要包含主诉、起病时间、主要症状、既往病史、当前用药、风险提示和建议科室。  
如果患者暂时不是急诊，还可以继续记录随访信息，比如体温变化、症状变化和当前用药，从而形成连续的病情追踪。”

### 6. 技术实现 5:10 - 6:00

建议话术：

“技术上，我们实现了一个完整的本地 MVP。  
后端基于 Node.js 和 Express，前端是轻量 Web 界面，核心分诊逻辑在 triage engine 中实现。  
同时我们补了单元测试、API 集成测试和 Playwright 端到端测试，确保主流程可以稳定运行。”

### 7. 价值与边界收尾 6:00 - 6:40

建议话术：

“CareBridge 的重点不是让 AI 替医生判断疾病，而是让更多患者在见到医生之前，就已经完成更清晰、更有质量的医疗准备。  
在医疗资源不均衡的现实下，这种更早一步的帮助，本身就是有价值的创新。”

## 录制建议

- 分辨率：`1920x1080`
- 浏览器缩放：`100%`
- 演示前先清空无关窗口
- 优先录真实网页，不要只播 PPT
- 录制时避免停顿太久，鼠标移动尽量稳定

## 演示重点提醒

- 不要把产品描述成“自动诊断”
- 强调 `risk guidance`、`visit preparation`、`doctor handoff`
- 讲社会价值时，回到“基层 / 偏远地区 / 资源不足”场景
- 讲技术时，不要过度展开细节，重点是闭环和可实现性
