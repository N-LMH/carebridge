# CareBridge 下一阶段功能端到端方案

## 1. 目标

本方案为 CareBridge 当前版本设计下一阶段最值得落地的一组新功能，目标不是堆功能，而是把现有的三端能力真正闭环：

- 患者端：从一次性分诊升级为连续观察与就医准备
- 医生端：从查看病例升级为高效跟进与结构化沟通
- 管理端：从静态统计升级为服务时效与风险运营

本期建议聚焦一个统一主题：

`连续随访与协作闭环`

围绕这个主题交付 5 个功能包：

1. 智能复诊提醒
2. 随访触发重新分级
3. 医生预问诊模板
4. 症状时间线
5. 管理端 SLA 与预警

## 2. 为什么先做这组

这组功能和当前架构最匹配，复用度最高：

- 现有患者端已经有 `session` 和 `follow-up`
- 现有后端已经有分诊引擎、消息、状态字段
- 现有医生端已经有队列和会话详情
- 现有管理端已经有统计、队列、状态筛选

也就是说，这一组不是推倒重来，而是在已有能力上补足“持续管理”这一层。

## 3. 交付边界

### 本期包含

- 提醒规则设计
- 风险重评估流程
- 医生快捷问询模板
- 单病例时间线视图
- 管理端时效指标与超时队列
- 对应数据模型、API、页面与测试方案

### 本期不包含

- 短信/微信真实消息触达
- 第三方医院系统打通
- 多医生排班与复杂派单
- 临床级 AI 自动回复

## 4. 功能包设计

## 4.1 功能包 A：智能复诊提醒

### 目标

让患者在首次分诊后知道“什么时候应该回来补一次状态”，避免分诊结果变成一次性页面。

### 用户价值

- 患者不会忘记继续观察
- 家属更容易持续记录
- 医生能更快看到后续变化

### 核心规则

- `Level 1`：不生成提醒，只强调立即急诊
- `Level 2`：默认提醒 `6-12 小时` 内补记一次
- `Level 3`：默认提醒 `24 小时` 内补记一次
- `Level 4`：默认提醒 `24-48 小时` 内补记一次

### 页面变化

患者结果页新增：

- `建议下次记录时间`
- `为什么要再次记录`
- `一键开始随访记录`

患者会话页新增：

- `待补记录` 状态条
- `已超时未记录` 轻提示

### 数据模型

`sessions` 表新增：

- `next_follow_up_at`
- `follow_up_plan_status`，取值建议：`none | scheduled | completed | overdue`
- `last_reassessment_at`

### API

- `POST /api/sessions/:id/follow-up-plan`
  - 生成或刷新随访计划
- `GET /api/sessions/:id/follow-up-plan`
  - 获取当前提醒计划

### 验收标准

- 生成分诊结果时自动得到下一次建议记录时间
- 完成一次随访后，提醒状态自动更新
- 超过建议时间未记录时，会话页可见提醒状态

## 4.2 功能包 B：随访触发重新分级

### 目标

让随访不只是“存一条 note”，而是真正影响风险判断。

### 用户价值

- 症状恶化时自动升级建议
- 医生端能更快看到风险变化
- 管理端能识别高风险演进病例

### 触发条件

提交随访记录后，如果出现以下任一条件，自动触发重新评估：

- 体温高于上一次记录
- 新增胸痛、呼吸困难、意识改变等危险信号
- 患者主动选择“症状明显加重”
- 连续多次随访无改善

### 输出结果

重新评估后，系统可产生：

- `riskLevel` 变化
- `actionLabel` 变化
- `redFlags` 变化
- `reassessmentDelta` 说明

示例：

- `Level 4 -> Level 2`
- `新增危险信号：高热伴加重`

### 页面变化

患者端：

- 随访提交后显示 `风险已更新`
- 结果页显示 `较上次变化`

医生端：

- 会话卡片新增 `风险升级` 标记
- 时间线中展示 `系统重评估事件`

管理端：

- 新增 `近期升级病例` 队列

### 数据模型

新增 `reassessments` 实体：

- `id`
- `session_id`
- `source`，如 `follow_up`
- `previous_risk_level`
- `new_risk_level`
- `delta_summary`
- `created_at`

### API

- `POST /api/sessions/:id/reassess`
  - 手动或系统触发重评估
- `GET /api/sessions/:id/reassessments`
  - 获取重评估历史

### 验收标准

- 符合规则的随访提交后自动触发重评估
- 风险升级会同步反映到患者端、医生端、管理端
- 能查询某个病例的所有重评估历史

## 4.3 功能包 C：医生预问诊模板

### 目标

减少医生重复输入，提高患者补信息的结构化程度。

### 模板分类

- 发热模板
- 呼吸道模板
- 胸痛模板
- 腹泻/呕吐模板
- 神经系统模板
- 通用复诊模板

### 模板示例

`胸痛补充信息`

- 疼痛持续多久？
- 是否向左臂、后背或下颌放射？
- 是否和活动有关？
- 是否伴呼吸困难、出汗或恶心？

### 页面变化

医生会话详情页新增：

- `快捷提问` 区块
- `插入模板` 按钮
- `发送前可编辑`

### 数据模型

可先做静态配置，不必第一期落库。

如需落库，可加：

- `message_templates`
  - `id`
  - `role_scope`
  - `category`
  - `title`
  - `content`

### API

第一期可不单独建后端 API，由前端静态模板驱动。

若需要后台管理模板：

- `GET /api/doctor/message-templates`

### 验收标准

- 医生可一键插入模板内容
- 模板发送前可修改
- 模板不替代自由输入，只是加速器

## 4.4 功能包 D：症状时间线

### 目标

把当前分散在初诊、随访、消息、状态变更里的信息，统一成一个病例时间线。

### 时间线事件类型

- `triage_created`
- `follow_up_added`
- `patient_message_sent`
- `doctor_message_sent`
- `reassessment_created`
- `doctor_status_changed`
- `admin_status_changed`

### 页面变化

患者会话页：

- 展示“我这次症状是怎么变化的”

医生端详情页：

- 展示“病例从创建到当前的关键节点”

管理端详情页：

- 展示“审核与处理轨迹”

### 数据实现建议

第一期不必新建完整事件总表，可以在后端聚合现有数据生成统一 timeline。

返回结构：

- `id`
- `type`
- `timestamp`
- `title`
- `description`
- `actor`
- `metadata`

### API

- `GET /api/sessions/:id/timeline`
- `GET /api/doctor/sessions/:id/timeline`
- `GET /api/admin/sessions/:id/timeline`

### 验收标准

- 单个病例的关键事件按时间倒序展示
- 风险变化、状态变化、消息往来都可在同一视图中看到
- 医生和管理员看到的时间线重点不同，但基于同一事件源

## 4.5 功能包 E：管理端 SLA 与预警

### 目标

让管理后台从“看总量”升级成“看时效、看积压、看升级风险”。

### 核心指标

- 高风险病例首次查看时长
- 高风险病例首次回复时长
- 超过 SLA 未处理病例数
- 24 小时内升级病例数
- 等待医生回复超时病例数

### 建议 SLA

- `Level 1`：5 分钟内进入处理视图
- `Level 2`：30 分钟内进入处理视图
- `等待医生回复`：2 小时内有首次响应

### 页面变化

管理首页新增：

- `SLA metrics strip`
- `超时病例队列`
- `风险升级病例队列`

管理详情页新增：

- `首次查看耗时`
- `首次回复耗时`
- `是否超 SLA`

### 数据模型

基于现有字段可先计算：

- `reviewed_at`
- `last_patient_message_at`
- `last_doctor_message_at`
- `created_at`

必要时新增：

- `first_doctor_response_at`
- `first_admin_review_at`

### API

- `GET /api/admin/sla`
- `GET /api/admin/queues?type=overdue_reply`
- `GET /api/admin/queues?type=risk_upgraded`

### 验收标准

- 管理端可看到 SLA 指标和超时队列
- 高风险升级病例能被单独筛出
- 能区分“已创建未处理”和“患者在等回复”两种不同积压

## 5. 端到端用户流程

### 患者

1. 完成首次分诊
2. 获得风险结果与建议复诊时间
3. 到达提醒时间后补充随访
4. 若病情加重，系统自动重评估
5. 若升级，则建议更快线下就医并同步推送医生侧高优先级

### 医生

1. 在工作台看到新增病例或升级病例
2. 打开病例详情查看时间线与最新风险
3. 通过快捷提问模板向患者补充关键信息
4. 根据补充信息更新状态、备注与优先级

### 管理员

1. 在后台查看高风险、超时、升级病例队列
2. 关注 SLA 超时病例
3. 审核病例流转是否顺畅
4. 对异常积压或高风险病例进行运营标记

## 9. 实施顺序

### Phase 1：最小闭环

- 智能复诊提醒
- 随访触发重新分级
- 患者端风险变化展示

这是最关键的业务闭环，建议最先做。

### Phase 2：医生效率提升

- 医生预问诊模板
- 医生端风险升级标记
- 医生端时间线

### Phase 3：运营可控性

- 管理端 SLA
- 超时队列
- 升级病例队列

- 时间线聚合逻辑

### API 测试

- 患者完成分诊 -> 看到提醒 -> 提交随访 -> 风险升级

- 患者获得复诊提醒

## 12. 成功标准

如果这一期做完，项目会从“能做一次分诊演示”升级成“能演示持续追踪与协作闭环”的产品：

- 患者端更有连续性
- 医生端更有工作台价值
- 管理端更像真实运营后台
- Demo 故事线更完整，也更像真实产品

## 13. 一句话结论

下一阶段最值得投入的不是继续加更多症状，而是把现有分诊结果做成一个可持续跟进、可升级、可协作的闭环系统；这会让 CareBridge 的产品完成度和说服力同时上一个台阶。
